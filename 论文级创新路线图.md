**从项目到论文：当前基础与后续路线**

本文档整理当前项目已经具备的研究基础，并给出继续向论文方向推进时更值得投入的几条路线。

**一、当前已经具备的基础**

| 维度 | 当前状态 |
|------|----------|
| 问答 baseline | 已补齐 `direct_llm`、`base_rag`、`improved_rag`、`problem_solving_rag` |
| 问答评测 | 已有结构化题集、逐题判分、冲突词检测 |
| 仿真 baseline | 已补齐 `rag`、`direct_llm`、`fixed` |
| 稳定性 | 已有多 seed 汇总和稳定种子控制 |
| 可视化 | 已生成 15 张图 |
| 汇报材料 | 已有 `showcase_summary.txt` 和归档目录 |

这说明项目已经具备整理成实验型短文初稿的基础材料。

**二、最值得继续深入的方向**

**方向 1：扩展问答数据集**

当前题集只有 8 道题，已经足够做课程汇报，论文阶段还需要更大规模的数据。

建议继续扩展到 30 到 50 道题，覆盖：

- 流程知识
- 传感器知识
- 故障排查
- 多参数联合约束
- 长尾材料与工件类型

扩展题集后，`improved_rag` 和 `problem_solving_rag` 的差异会更容易体现。

**方向 2：结构化知识库**

当前知识库仍是文本条目。下一步可以把知识组织成更明确的结构：

- 工件类型
- 材料属性
- 推荐夹爪力范围
- 推荐接近高度
- 典型注意事项

这样做之后，检索、参数选择和错误分析都会更清晰。

**方向 3：学习型参数模块加强**

仓库里已经有 `rag_learned` 路径，后续可以继续提高这一块的研究价值：

- 扩展训练样本
- 加入更多输入特征
- 研究 `uncertainty_std` 与成功率的关系
- 对比规则模块、LLM 模块和学习模块

**方向 4：仿真任务继续扩展**

当前 benchmark 已经有 train、val、test 划分。继续推进时，建议扩展：

- 更多材料组合
- 更多尺寸组合
- 多物体干扰任务
- 更细粒度的放置要求

这样可以把论文中的泛化实验做得更完整。

**方向 5：更接近真实系统**

如果后续有条件接触真实机械臂，可以保留现在这套仿真 benchmark 作为预实验，再补少量真实场景验证。这样整套工作会更有说服力。

**三、实验设计建议**

论文阶段建议保留以下四组核心实验。

问答实验：

- `direct_llm`
- `base_rag`
- `improved_rag`
- `problem_solving_rag`

仿真实验：

- `rag`
- `direct_llm`
- `fixed`

扩展实验：

- `rag_llm`
- `rag_learned`
- `rag_feedback`
- 检索消融 `rag / rag_multi / rag_random / fixed`

稳定性实验：

- 固定 `n_trials`
- 固定 `seeds`
- 报告 mean ± std

**四、当前最适合写进论文的亮点**

- 用同一份机械知识库同时驱动问答和仿真
- 问答部分已经有从直接 LLM 到问题解决型 RAG 的完整链路
- 仿真部分已经有 baseline、时间差异、多 seed 统计和可视化
- 结果组织方式已经比较规范，适合直接转成论文中的表格和图

**五、当前最需要补强的点**

- 题集规模还偏小
- 学习型模块还可以继续做得更完整
- 仿真任务数量还可以继续扩大
- 公开 benchmark 的说明文档还可以再整理得更规范

**六、推荐推进顺序**

| 阶段 | 建议内容 |
|------|----------|
| P0 | 继续稳定当前主流程，保证每次运行结果一致 |
| P1 | 扩展问答题集和仿真任务集 |
| P2 | 加强结构化知识库与检索重排 |
| P3 | 深化 `rag_learned` 和 `rag_llm` 的实验 |
| P4 | 整理匿名版代码与 benchmark 文档 |

**七、汇报与写作建议**

汇报时建议围绕这三句话展开：

- 机械知识库可以提升机械问答的稳定性
- 机械知识库可以提升抓取参数选择的成功率
- 多 seed、baseline 和图表让结果更容易说明

如果后续转成论文，标题方向可以围绕以下关键词组织：

- Knowledge-Guided Manipulation
- Retrieval-Augmented Parameter Selection
- Mechanical RAG Benchmark
